# Puter — Project overview

This document summarizes the repository layout, the project's high-level purpose, and how it integrates with AWS services. It focuses on the backend/kernel (`src/backend`) where most cloud integrations live.

## High-level summary

- Purpose: Puter is a modular desktop/web platform (kernel + extensions) with a backend that provides services such as file storage, drivers for AI and OCR, a web API, and a flexible module system. The codebase is split into documentation (`doc/`), modules and extensions (`mods/`, `mod_packages/`, `extensions/`), the main source tree (`src/`), tools and test suites (`tools/`, `test/`), and assorted configuration and CI files at the repository root.
- Backend: The backend kernel is located under `src/backend` and contains the service framework, filesystem and storage providers, APIs, and driver modules (for example, TTS and OCR). It uses an internal service/event system and integrates with a KV store, cost accounting, and filesystem abstractions.
- Storage and filesystem: Puter implements a virtual filesystem (PuterFS) with pluggable providers. Files may be stored in S3-backed storage; the filesystem layer passes `s3-info` and streaming interfaces so services can read directly from S3 when available.
- Modularity: The project organizes features as modules/drivers (`puterai`, `puterfs`, etc.) and exposes driver interfaces (e.g., `puter-tts`, `puter-ocr`) implemented by service classes.

## Where to look (important files / folders)

- `src/backend/` — backend services, routers, filesystem, drivers, and service registration.
- `src/backend/package.json` — dependencies including AWS clients and other third-party libraries.
- `src/backend/src/modules/puterai/` — modules that implement AI-related drivers (Polly, Textract, OpenAI wrappers).
- `src/backend/src/services/` — shared services such as SNS handling, Secrets Manager populator, and generic BaseService implementations.
- `src/backend/src/modules/puterfs/` — PuterFS provider and storage glue code (S3 integration points).

## AWS integration (detailed findings)

The repository integrates with several AWS managed services. The code uses AWS SDK clients to call managed APIs (Polly, Textract, Secrets Manager, SNS) and integrates with S3-backed file storage. There is no code in this repository that provisions, starts, stops, or terminates EC2 instances.

Summary of AWS services used

- AWS Secrets Manager
	- File: `src/backend/src/services/AWSSecretsPopulator.js`
	- What it does: Fetches a secret (named `puter-secrets`) using `@aws-sdk/client-secrets-manager` and overlays secret values into the app configuration. The populator looks for placeholders like `$__AWS_SECRET__` in config and replaces them with values from the secret string.
	- Evidence: constructs `SecretsManagerClient` and sends a `GetSecretValueCommand`, then JSON-parses `response.SecretString`.

- Amazon SNS
	- File: `src/backend/src/services/SNSService.js`
	- What it does: Exposes an HTTP endpoint `/sns` to receive SNS messages. It validates certificate URLs and signatures, confirms subscriptions, and forwards validated messages to the internal event bus.
	- Evidence: uses the v2 `aws-sdk` (`const aws = require('aws-sdk')`) and implements signature verification and certificate fetching logic.

- Amazon Polly (Text-to-Speech)
	- File: `src/backend/src/modules/puterai/AWSPollyService.js`
	- What it does: Lists available Polly voices and synthesizes speech via `@aws-sdk/client-polly`. It performs cost checks via the internal cost service before calling Polly, caches voice lists, and supports multiple engines (standard, neural, long-form, generative).
	- Evidence: constructs `PollyClient`, uses `DescribeVoicesCommand` and `SynthesizeSpeechCommand`. Credentials are created from `this.config.aws.access_key` and `this.config.aws.secret_key`.

- Amazon Textract (Document analysis / OCR)
	- File: `src/backend/src/modules/puterai/AWSTextractService.js`
	- What it does: Calls Textract `AnalyzeDocumentCommand` to perform OCR/layout extraction. It supports both S3 object references (S3Object) and in-memory bytes uploads, and enforces internal cost checks before running.
	- Evidence: constructs `TextractClient` with credentials from config; `_get_client_and_document` uses `s3-info` to form `S3Object` or falls back to `Bytes`.

- Amazon S3 (storage integration)
	- Files: `src/backend/src/services/drivers/FileFacade.js`, `src/backend/src/filesystem/ll_operations/ll_read.js`, `src/backend/src/modules/puterfs/lib/PuterFSProvider.js`, and other filesystem/storage glue code.
	- What it does: Supports streaming files directly from S3 (reads `s3:location`, `bucket`, `key`, `bucket_region`) and provides read/copy operations that delegate to storage providers which may be S3-backed.
	- Evidence: `FileFacade` exposes `s3-info` and can provide streams from S3; `ll_read` calls storage.create_read_stream with `{ bucket, bucket_region, key }`.

Authentication and configuration

- Credentials: Polly and Textract clients are created with credentials formed from `this.config.aws.access_key` and `this.config.aws.secret_key`, and region derived from `this.config.aws.region` or `this.global_config.aws.region` (with default `us-west-2` in code paths).
- Secrets Manager integration: `AWSSecretsPopulator` reads the `puter-secrets` secret and overlays secret values into the global configuration. This indicates secrets/credentials may be injected at runtime from AWS Secrets Manager when configured.

EC2 and instance management — explicit answer

- The repository does not contain any code that manages EC2 instances. Searches for EC2-related APIs and terms (e.g., `EC2Client`, `runInstances`, `startInstances`, `terminateInstances`, `describeInstances`, `InstanceId`, or AMI ids like `ami-`) returned no matches in the source. The AWS usage is limited to managed services (Polly, Textract, Secrets Manager, SNS) and S3-backed storage access.

What this implies

- The code interacts with AWS at the API level to use managed services and S3 storage. It expects AWS credentials and region to be provided via `this.config` (and optionally populated from Secrets Manager).
- Provisioning or lifecycle management of cloud compute (EC2) is not handled here; if EC2 instance lifecycle management is required for your deployment, it is either done outside this repository (deployment scripts, IaC in another repo, CI/CD, or manually) or not implemented.

## AI features and unified Chat Completion API

Puter exposes a single, unified AI/chat completion interface that abstracts multiple provider implementations and presents them as one API to clients. Key points:

- Core components
  - Orchestrator: src/backend/src/modules/puterai/AIChatService.js — central logic for routing requests, normalization, moderation, usage checks, streaming, fallbacks, and cost recording.
  - Interface/driver: src/backend/src/modules/puterai/AIInterfaceService.js — registers the `puter-chat-completion` driver with methods like `complete`, `models`, and `list`.
  - Module loader: src/backend/src/modules/puterai/PuterAIModule.js — registers provider services based on configuration.

- Provider adapters
  - Providers are implemented as pluggable services (examples: OpenAI, Claude, Mistral, OpenRouter). Each provider service registers itself and implements the driver `puter-chat-completion`.
  - Files: examples include src/backend/src/modules/puterai/OpenAICompletionService.js and src/backend/src/modules/puterai/ClaudeService.js.

- Contract and call flow
  - Client calls AIChatService.complete with messages, optional model, tools, and stream flag.
  - Messages are normalized, moderated, and tools are normalized for function-calling semantics.
  - AIChatService chooses intended provider/model (config-driven), checks funding via the cost service, then delegates the call to the provider adapter.
  - Providers return either a synchronous completion + usage info or a streaming TypedValue (`ai-chat-intermediate`) that the orchestrator pipes as NDJSON streaming responses.

- Streaming, tools, and files
  - Streaming is supported via provider-initiated stream handlers; AIChatService wraps these into streamed responses and ties them to usage promises.
  - Function-calling/tool normalization is implemented in src/backend/src/modules/puterai/lib/FunctionCalling.js.
  - Providers upload and manage temporary files when needed (provider-specific file APIs).

- Moderation, metering, and fallbacks
  - Content moderation runs before provider calls (helper: src/backend/src/modules/puterai/lib/AsModeration.js).
  - Usage/cost accounting is enforced: AIChatService queries the cost service and can route to a usage-limited fallback if funds are insufficient.
  - On provider failures the orchestrator attempts configured fallbacks (based on model cost/compatibility) before surfacing errors.

- Credentials and configuration
  - Provider credentials and region come from per-service config or global config; AWS secrets may be injected by AWSSecretsPopulator.
  - Providers are enabled/disabled and aliased entirely via configuration.

- Observability and extension
  - Events: AIChatService emits ai.prompt.report-usage and ai.prompt.cost-calculated events used by the cost/spending services.
  - Adding a new provider requires implementing the driver contract, registering the service, and handling provider-specific tool/file semantics.

Recommended quick improvements
- Add unit tests for non-streaming, streaming, and fallback flows.
- Add concise README snippet documenting config keys for providers and expected secret shapes (openai.apiKey, claude.apiKey, etc.).
